# Cupid SQL 최종 발표 대본 (5분 30초)

---

## 1. 인사 + 프로젝트 소개 (40초)

안녕하세요. Cupid SQL 프로젝트를 발표하겠습니다.

Cupid SQL은 **실제 C 언어로 만든 SQL 엔진**을 웹 브라우저에서 시각적으로 체험할 수 있는 시스템입니다.

이 프로젝트의 핵심은 웹이 SQL을 흉내내는 게 아니라, **진짜 C 엔진을 호출한다**는 점입니다. 사용자가 SQL을 입력하면 Node.js 백엔드가 C 엔진을 실행하고, 그 결과를 웹에 시각화합니다. 그리고 최종 서비스로는 **데이트 매칭 앱**을 구현했습니다.

---

## 2. 팀 역할 + 협업 구조 (50초)

저희 팀은 4명이 각각 하나의 패널을 담당했습니다.

- **A (세민)** - 패널 1, CLI. SQL을 입력하는 터미널 화면입니다. xterm.js로 실제 쉘 세션을 열고, 그 안에서 C 엔진을 실행합니다.
- **B (규고)** - 패널 2, Parse Tree. SQL 파싱 구조를 ReactFlow로 트리 시각화합니다.
- **C (규태, 저)** - 패널 3, Service. 데이트 매칭 서비스 화면입니다.
- **D (동현)** - 통합 담당. 백엔드, C 엔진, 그리고 전체 연결을 맡았습니다.

협업은 **공용 계약 문서**를 기반으로 했습니다. 요청/응답 형식과 테이블 스키마를 먼저 정해놓고, 각자 브랜치에서 독립적으로 개발한 뒤 dev에 PR로 병합했습니다.

---

## 3. 화면 시연 (1분 30초)

> (화면을 띄우며)

지금 보시는 게 3패널 통합 화면입니다.

**왼쪽이 CLI 패널**입니다. 이건 단순한 입력창이 아니라, 실제 터미널 세션입니다. WebSocket으로 백엔드에 연결되어 있고, C 엔진을 직접 실행합니다. 예제 SQL 버튼을 누르면 바로 쿼리가 실행됩니다.

**가운데가 Parse Tree 패널**입니다. SQL을 실행하면 파싱 구조가 노드와 엣지로 시각화됩니다. SELECT, FROM, WHERE 같은 키워드가 각각 노드로 표현돼서 SQL의 구조를 한눈에 볼 수 있습니다.

**오른쪽이 제가 만든 서비스 패널**, Cupid SQL 데이트 매칭입니다.

> (시연하며)

이름, MBTI, 취미를 입력하고 버튼을 누르면... 하트 로딩 애니메이션이 나오고... 이렇게 매칭 결과가 카드 형태로 나옵니다. 가장 잘 맞는 사람에게 BEST 뱃지가 붙고, 매칭률이 프로그레스 바로 표시됩니다. 매칭 점수에 따라 코멘트도 자동으로 달라집니다.

그리고 이 데이터는 실제로 C 엔진의 데이터베이스에 저장됩니다. 프로필을 등록하면 INSERT 쿼리가 실행되고, 매칭할 때는 SELECT로 전체 프로필을 가져와서 비교합니다.

---

## 4. 데이터 흐름 (1분)

SQL을 실행하면 내부적으로 이런 흐름이 일어납니다.

1. 사용자가 CLI에서 SQL을 입력하거나, 서비스 패널에서 프로필을 등록합니다.
2. WebSocket 또는 HTTP로 백엔드에 전달됩니다.
3. Node.js 백엔드가 C 엔진을 프로세스로 실행합니다.
4. C 엔진이 SQL을 파싱하고 실행해서 결과를 stdout으로 출력합니다.
5. 백엔드가 그 출력을 JSON으로 변환해서 프론트엔드에 보냅니다.
6. 프론트엔드가 응답을 받아서 3개 패널을 동시에 갱신합니다.

핵심은 **프론트엔드는 보여주기만 하고, 실제 데이터 처리는 C 엔진이 한다**는 점입니다. 데이터베이스 테이블은 profiles 테이블 하나이고, name, mbti, hobby 3개 컬럼으로 구성되어 있습니다.

---

## 5. 기술 스택 + 배운 점 (1분)

기술 스택은 프론트엔드가 React와 Vite, 트리 시각화에 ReactFlow, 터미널에 xterm.js를 썼습니다. 백엔드는 Node.js Express에 WebSocket을 사용하고, 엔진은 C로 gcc와 Makefile로 빌드합니다. 데이터는 파일 기반 CSV 저장이고, 협업은 Git과 GitHub PR을 활용했습니다.

개발하면서 배운 점 세 가지입니다.

첫째, **계약 문서의 중요성**입니다. 4명이 동시에 개발하는데, 요청/응답 형식만 맞추면 각자 독립적으로 작업할 수 있었습니다.

둘째, **Git 브랜치 전략**입니다. 역할별 브랜치를 분리하고 dev에 PR로 병합하면서, 실제로 충돌이 발생했을 때 어떻게 해결하는지 경험했습니다.

셋째, **시스템 통합의 어려움**입니다. 프론트엔드, 백엔드, C 엔진 세 계층이 각각 독립적으로 잘 동작해도, 연결하면 예상치 못한 문제가 생깁니다. 인코딩 에러, 플랫폼 호환성 같은 이슈를 직접 해결했습니다.

## 6. 마무리 (20초)

정리하면 Cupid SQL은, 진짜 C 엔진을 웹에서 호출하는 구조, 4명이 독립적으로 개발 가능한 아키텍처, 공용 계약 기반의 팀 협업, 그리고 SQL 결과를 데이트 매칭 서비스로 시각화한 프로젝트입니다.

감사합니다.
