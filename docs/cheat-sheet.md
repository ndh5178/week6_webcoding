# CUPID SQL 발표 컨닝페이퍼

---

## 1. 인사 + 소개

안녕하세요. Cupid SQL 프로젝트를 발표하겠습니다.

Cupid SQL은 **실제 C 언어로 만든 SQL 엔진**을
웹 브라우저에서 시각적으로 체험할 수 있는 시스템입니다.

핵심은 웹이 SQL을 흉내내는 게 아니라
**진짜 C 엔진을 호출한다**는 점입니다.

최종 서비스로는 **데이트 매칭 앱**을 구현했습니다.

---

## 2. 아키텍처

> 리드미 Architecture 다이어그램을 보여주며:

이 그림이 전체 구조입니다.

맨 위가 브라우저, React로 만든 3개 패널.
가운데가 Node.js 백엔드, 프론트와 엔진을 연결하는 다리.
맨 아래가 C 엔진, SQL을 실제로 처리하는 두뇌입니다.

사용자가 SQL을 입력하면
백엔드가 C 엔진을 실행하고
결과를 JSON으로 변환해서
브라우저 3개 패널에 동시에 보여줍니다.

---

## 3. 3패널 구조

> 리드미 3-Panel Layout을 보여주며:

화면은 3등분입니다.

**왼쪽 CLI 패널** → 실제 터미널. C 엔진을 직접 실행.
담당 김세민.

**가운데 Parse Tree** → SQL 구조를 트리로 시각화.
담당 김규민.

**오른쪽 서비스 패널** → 데이트 매칭 앱.
담당 김규태, 저입니다.

통합은 남동현이 맡았습니다.

---

## 4. 데이터 흐름

> 리드미 Data Flow를 보여주며:

데이터 흐름입니다.

사용자가 입력하면
브라우저에서 백엔드로 전달되고
백엔드가 C 엔진을 실행하고
C 엔진이 CSV 파일에서 데이터를 읽거나 저장하고
결과가 다시 브라우저로 돌아와서
3개 패널이 동시에 갱신됩니다.

**프론트엔드는 보여주기만 하고
실제 데이터 처리는 C 엔진이 합니다.**

---

## 5. 핵심 로직: SQL 파싱 흐름

> 리드미 Core Logic: SQL Parsing Flow를 보여주며:

이게 우리 프로젝트의 핵심 로직입니다.

SQL 명령이 들어오면 4단계를 거칩니다.

**main.c** → 입력을 받고
**parser.c** → SQL 텍스트를 분석해서 구조체로 변환합니다.
예를 들어 INSERT 문이면
테이블 이름이 뭔지, 값이 뭔지를 분리합니다.

**executor.c** → 분석된 구조체를 받아서 실행합니다.
schema.c에서 테이블 구조를 확인하고

**storage.c** → 실제 CSV 파일에 저장하거나 읽습니다.

INSERT면 CSV 파일 끝에 한 줄 추가.
SELECT면 CSV 파일을 한 줄씩 읽으면서
WHERE 조건에 맞는 행만 골라서 반환합니다.

결과는 stdout으로 출력되고
백엔드가 이걸 캡처해서 JSON으로 변환합니다.

---

## 6. 시연

> 웹사이트를 띄우며:

실제 화면을 보여드리겠습니다.

**CLI에서 INSERT 실행** →
프로필이 등록되고 실제로 vs코드에서 profiles.csv파일에 데이터가 추가되고 Parse Tree에 구조가 시각화됩니다.

> 가운데 패널을 가리키며:

지금 가운데 트리를 보시면
INSERT가 루트 노드이고
그 아래 INTO와 VALUES로 나뉩니다.
INTO 아래에는 테이블 이름 profiles,
VALUES 아래에는 입력한 값들이
각각 하나의 노드로 표현됩니다.

이게 아까 말한 parser.c가 SQL을 분석한 결과입니다.
SQL 한 줄이 실제로는 이런 트리 구조로 분해된다는 걸
시각적으로 보여주는 게 이 패널의 역할입니다.

**SELECT 실행** →
저장된 데이터가 조회되고 트리 구조도 바뀝니다.
SELECT 노드 아래에 COLUMNS, FROM이 나오고
WHERE 조건을 붙이면 WHERE 노드도 추가됩니다.

**서비스 패널에서 매칭** →
이름, MBTI, 취미 입력하고 버튼 누르면
하트 로딩 후 매칭 결과가 카드로 나옵니다.
BEST 뱃지, 매칭률 바, 코멘트가 자동 표시됩니다.

이 데이터는 C 엔진 DB에 실제로 저장되고
매칭할 때는 SELECT로 전체를 가져와서 비교합니다.

## 7. 문제점
규민: Parse Tree를 처음에는 컴포넌트로 직접 트리 노드를 구현했는데
ReactFlow 라이브러리를 도입하면서 훨씬 간편하게 개발할 수 있었습니다.
적절한 라이브러리 선택이 개발 효율을 크게 높인다는 걸 알게 되었습니다.

세민: CLI 패널을 만들 때 처음에는 input 태그로 명령어를 받는 방식이었는데,
이러면 진짜 터미널처럼 보이지 않는 문제가 있었습니다.
그래서 xterm.js 라이브러리를 도입해서
실제 shell 프로세스를 연결하고
브라우저 안에서 진짜 터미널을 띄우는 방식으로 해결했습니다.

규태: 서비스 패널을 만들 때 처음에는 단독 페이지로 개발했는데
3패널 통합 화면에 넣으니 레이아웃이 깨지는 문제가 있었습니다.
패널 크기에 맞게 스타일을 전부 다시 조정해야 했고
독립적으로 만든 컴포넌트를 통합 환경에 맞추는 과정이 생각보다 까다로웠습니다.

동현: 통합 과정에서 브랜치 전략이 미흡해서
병합할 때 충돌이 많이 발생했고
충돌을 해결하는 과정에서 기능이 상당 부분 날아갔습니다.
결국 전부 초기화하고 처음부터 다시 통합했습니다.
이 경험을 통해 브랜치 전략과 충돌 관리의 중요성을 체감했습니다.

---
## 8. 마무리

정리하면 Cupid SQL은
**진짜 C 엔진**을 웹에서 호출하는 구조,
SQL이 입력부터 저장까지 **4단계 파싱 흐름**을 거치고,
4명이 **독립적으로 개발** 가능한 아키텍처,
그리고 SQL 결과를 **데이트 매칭 서비스**로 시각화한 프로젝트입니다.

감사합니다.

## Q&A 답변

**왜 C로?**
→ 시스템 프로그래밍 레벨에서 SQL 파서와 실행기를 직접 구현하는 것이 학습 목표

**매칭 알고리즘?**
→ MBTI 4글자 비교, 글자당 20점 + 취미 같으면 20점 = 최대 100점, 상위 3명

**데이터 저장?**
→ C 엔진이 CSV 파일로 저장. MySQL 안 씀

**충돌?**
→ 역할별 브랜치 분리 + dev에 PR 병합. 담당 영역 명확해서 바로 판단

**테이블 구조?**
→ profiles 테이블 하나. name, mbti, hobby 3개 컬럼

**API 계약이 뭐에요?**
→ 팀원끼리 "이 형식으로 보내고 받자" 미리 약속한 문서. 이게 있어서 4명이 동시에 개발 가능

**파싱이 뭐에요?**
→ SQL 텍스트를 컴퓨터가 이해할 수 있는 구조로 분해하는 과정. parser.c가 담당
