# CUPID SQL 발표 컨닝페이퍼 (리드미 기반)

---

## 1. 인사 + 소개

안녕하세요. Cupid SQL 프로젝트를 발표하겠습니다.

Cupid SQL은 **실제 C 언어로 만든 SQL 엔진**을
웹 브라우저에서 시각적으로 체험할 수 있는 시스템입니다.

핵심은 웹이 SQL을 흉내내는 게 아니라
**진짜 C 엔진을 호출한다**는 점입니다.

최종 서비스로는 **데이트 매칭 앱**을 구현했습니다.

> 리드미의 Architecture 다이어그램을 보여주며:

이 그림이 전체 구조입니다.
위에 브라우저, 가운데 Node.js 백엔드, 아래 C 엔진.
사용자가 SQL을 입력하면 백엔드가 C 엔진을 실행하고
결과를 웹에 시각화합니다.

---

## 2. 팀 역할

> 리드미의 Team 표를 보여주며:

저희 팀은 4명이 각각 하나의 패널을 담당했습니다.

- **김세민** → 패널 1, CLI. 터미널 화면.
  xterm.js로 실제 쉘 세션을 열고 C 엔진을 실행
- **김규민** → 패널 2, Parse Tree.
  SQL 파싱 구조를 ReactFlow로 트리 시각화
- **김규태, 저** → 패널 3, Service.
  데이트 매칭 서비스 화면
- **남동현** → 통합 담당.
  백엔드, C 엔진, 전체 연결

> 리드미의 API Contract를 가리키며:

이 API 계약 문서를 먼저 정해놓고
각자 브랜치에서 독립적으로 개발한 뒤
dev에 PR로 병합했습니다.

---

## 3. 화면 시연

> 리드미의 3-Panel Layout을 가리키며:

이게 저희 3패널 구조입니다.
실제 화면을 보여드리겠습니다.

> 웹사이트를 띄우며:

**왼쪽 CLI 패널** → 단순한 입력창이 아니라 실제 터미널.
C 엔진을 직접 실행합니다.

**가운데 Parse Tree 패널** → SQL 실행하면
SELECT, FROM, WHERE 같은 키워드가 노드로 시각화됩니다.

**오른쪽 서비스 패널** → 제가 만든 데이트 매칭입니다.

> 직접 시연:

이름, MBTI, 취미를 입력하고 버튼을 누르면
하트 로딩 후 매칭 결과가 카드로 나옵니다.
BEST 뱃지, 매칭률 바, 코멘트가 자동으로 표시됩니다.

이 데이터는 C 엔진 DB에 실제로 저장됩니다.
등록하면 INSERT, 매칭할 때는 SELECT가 실행됩니다.

---

## 4. 데이터 흐름

> 리드미의 Data Flow를 가리키며:

이 그림이 데이터 흐름입니다.

1. 사용자가 SQL 입력하거나 프로필 등록
2. C 엔진이 SQL을 파싱하고 실행해서 결과 출력
3. 프론트엔드가 응답을 받아서 3개 패널 동시 갱신

핵심은 **프론트엔드는 보여주기만 하고
실제 데이터 처리는 C 엔진이 한다**는 점입니다.

> 리드미의 Database Schema를 가리키며:

테이블은 profiles 하나.
name, mbti, hobby 3개 컬럼입니다.

---

## 5. 매칭 알고리즘 + 배운 점

> 리드미의 Match Algorithm을 가리키며:

매칭 알고리즘입니다.
MBTI 4글자를 하나씩 비교해서 글자당 20점.
취미가 같으면 추가 20점. 최대 100점.

개발하면서 배운 점 두 가지입니다.

첫째, **계약 문서의 중요성**.
4명이 동시에 개발하는데
요청/응답 형식만 맞추면 독립적으로 작업 가능했습니다.

둘째, **시스템 통합의 어려움**.
프론트, 백엔드, C 엔진이 각각 잘 동작해도
연결하면 예상치 못한 문제가 생깁니다.
인코딩 에러, 플랫폼 호환성 이슈를 직접 해결했습니다.

---

## 6. 마무리

정리하면 Cupid SQL은
진짜 C 엔진을 웹에서 호출하는 구조,
4명이 독립적으로 개발 가능한 아키텍처,
공용 계약 기반의 팀 협업,
SQL 결과를 데이트 매칭 서비스로 시각화한 프로젝트입니다.

감사합니다.

## Q&A 답변

**왜 C로?**
→ 시스템 프로그래밍 레벨에서 SQL 파서와 실행기를 직접 구현하는 것이 학습 목표

**매칭 알고리즘?**
→ MBTI 4글자 비교, 글자당 20점 + 취미 같으면 20점 = 최대 100점, 상위 3명

**데이터 저장?**
→ C 엔진이 CSV 파일로 저장. MySQL 안 씀

**충돌?**
→ 역할별 브랜치 분리 + dev에 PR 병합. 담당 영역 명확해서 바로 판단

**테이블 구조?**
→ profiles 테이블 하나. name, mbti, hobby 3개 컬럼

**API 계약이 뭐에요?**
→ 팀원끼리 "이 형식으로 보내고 받자" 미리 약속한 문서. 이게 있어서 4명이 동시에 개발 가능
